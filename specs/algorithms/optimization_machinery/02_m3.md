# M3 (Multi-Scale Momentum with CMS)

```
CONTRACT
  Purpose:    CMS applied to the optimizer itself. Standard momentum operates
              at a single timescale. M3 uses k frequency levels, each with
              its own momentum accumulator at a different timescale.
              Fast levels track short-term gradient trends.
              Slow levels track long-term optimization direction.
  Expects:    Gradient signal. k frequency levels with chunk sizes.
              Per-level momentum state. Pulse with active levels.
  Guarantees: Multi-scale gradient history compression.
              Level 0 (fastest) captures per-token patterns.
              Level k-1 (slowest) captures sequence-wide trends.
              Frozen levels accumulate error for later application.
  Cost:       k * O(per_level_momentum_cost). But only active levels compute.
              At most steps, only level 0 is active (cheapest).
  Trade-off:  More levels = finer temporal resolution = more state.
              k=1 recovers standard single-scale momentum.
              k=4 with [1,8,64,512] frequencies gives 4 temporal scales.
  Position:   specs/algorithms/optimization_machinery/02_m3.md
  Source:     HOPE (2512.24695) Eq 75 (M3 two-level momentum), Eq 42 (Muon),
              Eq 44 (Newton-Schulz iteration)
```

## Paper-to-Code Reconciliation

Eq 75 LaTeX shows `M² = M² - beta_2 * sum(g)` (subtraction on slow momentum).
The HOPE reference implementation (`M3Optimizer.step()` in
`HOPE/src/hope/m3_optimizer.py`) uses **addition** for the slow momentum update:
`M2.mul_(beta3).add_(grad, alpha=1.0 - beta3)`. Both M¹ and M² are additive EMA.
Our implementation matches the reference code, not the LaTeX.

The multi-scale benefit comes from temporal resolution (M² updates only every Ĉ
steps, producing a smoother signal), not from directional opposition. This is
consistent with how CMS levels work in the memory system — different frequencies,
same direction.

## Architecture

```
-- Standard momentum: one accumulator, one timescale
S = eta * S + theta * grad

-- M3: k accumulators at k timescales
FOR level = 0 to k-1:
  IF pulse.is_active(level):
    S[level] = eta[level] * S[level] + theta[level] * accumulated_grad[level]
    accumulated_grad[level] = zeros()  -- reset after application
  ELSE:
    accumulated_grad[level] += grad    -- accumulate for later

-- The COMBINED momentum is a weighted sum across levels:
S_combined = sum(weight[level] * S[level] for level in 0..k)
```

## Update Path Split (2D vs 1D)

The HOPE reference implementation splits the parameter update into two paths:

**2D params** (weight matrices: W_Q, W_K, W_V, W_O, W_embed, W_unembed):
- Each momentum buffer (M¹, M²) is NS-orthogonalized independently
- Combined update: `param -= lr * (NS(M¹) * ‖M¹‖ + alpha * NS(M²) * ‖M²‖)`
- No per-element second moment V — NS is a global operation that fills all
  elements, and dividing by sqrt(V) would amplify near-zero-gradient entries

**1D params** (biases, layer norms: b_alpha, b_theta, ln_gamma, ln_beta):
- Raw momentum used without NS (NS requires 2D reshape)
- Adam-style V division: `param -= lr * (M¹ + alpha * M²) / (sqrt(V_hat) + eps)`
- V_hat = V / (1 - beta2^t) is bias-corrected

This split is the key difference from standard Muon, which applies NS uniformly.

**Memory**: 3 buffers per param (M¹ + M² + V) in param dtype. At bf16 this is
6 bytes/param — less than AdamW's 2 × fp32 = 8 bytes/param.

## Pseudocode

```text
ALGORITHM: m3_step(params, grads, state, config, step)
  beta1, beta2, beta3 = config.betas
  alpha, lr, eps = config.alpha, config.lr, config.eps

  FOR each (param, grad) in (params, grads):
    -- Step 1: Update fast momentum (every step)
    M1 = beta1 * M1 + (1 - beta1) * grad

    -- Step 2: Update second moment (every step)
    V = beta2 * V + (1 - beta2) * grad^2

    -- Step 3: Update slow momentum (every Ĉ steps)
    IF step % chunk_size == 0:
      M2 = beta3 * M2 + (1 - beta3) * grad

    -- Step 4: Apply update (split by param shape)
    IF param.dim >= 2:
      -- Muon-style: NS orthogonalize each momentum independently
      O1 = NS(reshape_2d(M1)) * norm(M1)
      O2 = NS(reshape_2d(M2)) * norm(M2)
      param -= lr * (O1 + alpha * O2)
    ELSE:
      -- Adam-style: V division with bias correction
      V_hat = V / (1 - beta2^step)
      param -= lr * (M1 + alpha * M2) / (sqrt(V_hat) + eps)
```

### Newton-Schulz Orthogonalization (Eq 44)

```text
FUNCTION: NS(M, iterations=5) -> Tensor
  -- Polynomial form (Muon coefficients: a=3.4445, b=-4.7750, c=2.0315)
  -- Transpose tall matrices: work on min(rows,cols) dimension
  X = M / norm(M)
  FOR i = 1 to iterations:
    A = X @ X^T
    X = a*X + b*(A @ X) + c*(A @ (A @ X))
  return X
```

## Example: k=2 Levels

```
-- Level 0: frequency 1 (updates every step)
--   Captures per-token gradient patterns
--   Fast, reactive, noisy
--   eta[0] = 0.9 (short memory)

-- Level 1: frequency 8 (updates every 8 steps)
--   Captures multi-token trends
--   Slow, stable, smooth
--   eta[1] = 0.99 (long memory)

-- Between updates, level 1 accumulates 8 gradients.
-- When it fires, it applies the ACCUMULATED gradient — smoother signal.
-- This is analogous to how CMS operates on model parameters,
-- but applied to the optimizer's own state.
```

## Connection to CMS

M3 is CMS applied recursively:

```
CMS on model params:   param[level] updates at frequency[level]
M3 on optimizer state:  momentum[level] updates at frequency[level]

-- CMS + M3: both model AND optimizer are multi-scale
-- This is NL IS #2 taken to its logical conclusion:
-- nested, multi-level, parallel optimization
-- at EVERY level of the system
```

## Axiom Compliance

- **NL IS #2** (nested, multi-level): k optimization levels at k frequencies
- **NL IS #8** (continuum memory): Discrete approximation of continuous temporal scales
- **NL IS #6** (optimizers are memory): Each momentum level IS a memory at its timescale
- **NL IS NOT #1** (not single-level): k levels, each independent
